# 전용 업무 앱 스펙 v2 (WORK-APP-SPEC-v2)

상단 탭 5~6개로 확장 예정이며, **탭을 하나씩 순차 구축**한다.
현재 단계: **1번 탭 '천하통일'(인트라넷 브라우저)만 구현.** 나머지 탭은 자리만 잡아두고 "준비 중" 표시.

기존 CONTRACT-APP-SPEC(계약서 전자서명)을 마지막 단계로 흡수하는 상위 앱 확장.
기존 계약서 코드는 수정 없이 유지하고, 앞단에 고객·매물 관리 층을 추가한다.

## 전체 플로우 (6단계)
```
① 앱 로그인 → 인트라넷 → ② 매물카드(전화번호 조회) → ③ 반영/수수료 계산
→ ④ 메모 → ⑤ 계약 및 결제(계약서 자동 주입) → ⑥ 서명·PDF·공유·바로결제(기존)
```

## ① 로그인 및 인트라넷
- 앱 자체 잠금: PIN 로그인 (기존 스펙 유지)
- 로그인 직후 화면: **인트라넷 탭** (기본 첫 화면)
  - **1차 구현**: iframe 임베드로 인트라넷 로그인 페이지 표시. 사용자가 직접 로그인 → 세션 쿠키는 인트라넷 서버와 브라우저 간 직접 처리 (앱은 자격증명에 관여하지 않음)
  - **구현 전 테스트 (필수, 5분)**: 단순 HTML에 iframe으로 인트라넷 URL 삽입 → iPad Safari에서 (a) 표시 여부(X-Frame-Options/CSP 차단 확인) (b) 로그인 후 세션 유지 여부(Safari 서드파티 쿠키 차단 확인)
  - **폴백**: 둘 중 하나라도 실패 시 대시보드에 "인트라넷 열기" 버튼 → `window.open` 새 탭. iPad Split View 병행 사용 안내
- **상단 탭 내비게이션 (5~6개 예정)**: [천하통일] [요약] [노트] [계약서] [+예정]
  - 천하통일(1번 탭): 인트라넷 브라우저. 임베드 성공 시 웹뷰(뒤로가기·새로고침 포함), 실패 시 "인트라넷 열기" 새 탭 버튼
  - **다중 창 (천하통일 탭 내부)**:
    - 탭 바 아래 창 칩 스트립: (창1) (창2) ... [+], 탭하면 전환, X로 닫기, 길게 눌러 이름 변경
    - 각 창 = 독립 iframe, 전환 시 숨김 처리만(unmount 금지) → 스크롤·입력·세션 상태 유지
    - 창 개수 상한 5개 (메모리 보호)
    - 알려진 한계: cross-origin이라 창 제목 자동 표시 불가(수동 이름), 인트라넷 popup(window.open)은 스트립으로 포섭 불가 — popup 다용 시 임베드 방식 재검토
  - 요약 / 노트: AI 기능 층 참조 (아래 섹션)
  - 계약서: 기존 계약서 작성 플로우

## AI 기능 층 (요약 탭 · 노트 탭 공통)
- **인프라**: Anthropic API 사용. API 키는 클라이언트에 노출 금지 — Vercel Serverless Function(/api/ai)을 프록시로 두고 키는 환경변수 보관. 모델: claude-sonnet-4-6 (요약·노트 용도 충분, 비용 효율)
- **② 요약 탭**
  - 인트라넷 자동 크롤링 불가 (인증 페이지 cross-origin 접근은 브라우저가 차단) → **붙여넣기 기반**
  - UI: 큰 텍스트 붙여넣기 영역 + [요약] 버튼 + 프리셋 3종(매물 목록 → 조건 표 정리 / 공지·문서 → 3줄 요약 / 자유 요약)
  - 결과는 복사 버튼 제공, 필요 시 노트로 저장
- **③ 노트 탭 (대화형 노트)**
  - 노트 = 대화 스레드. 메시지 입력 → Claude 응답 → 전체가 노트로 저장
  - notes 테이블: id, listing_id(FK, nullable — 매물카드 연결 선택), title, messages(jsonb), updated_at
  - 매물카드에서 [노트] 진입 시 해당 카드 정보(상호, 조건)를 대화 컨텍스트로 자동 포함
  - 시스템 프롬프트: 상가 창업·점포 거래 상담 보조 컨텍스트 고정

## ② 매물카드 (고객·매물 관리)
- 진입: 전화번호 입력 (숫자 키패드, 010-0000-0000 자동 포맷)
  - 기존 고객 → 매물카드 즉시 오픈
  - 신규 → 새 카드 생성 화면
- 카드 기본 정보:
  - 고객: 전화번호(key), 고객명, 구분(양도자/임차인/기타)
  - 매물: 상호, 업종(기존 업종 선택 UI 재사용), 사업자등록번호, 소재지
  - 거래 조건: 보증금, 월세, 권리금, 관리비
- 목록 화면: 최근 상담순 정렬, 전화번호/상호 검색

## ③ 반영 → 수수료 계산 화면
- 매물카드에서 [반영] 버튼 → 계산 결과 화면 로딩
- 자동 계산 항목:
  - **권리금 수수료**: 권리금 × 3~5% (VAT 별도) — 3/4/5% 슬라이더 또는 버튼, VAT 포함가 병기
  - **부동산 중개보수**: (보증금 + 월세×100) × 0.9% (VAT 별도)
  - 광고료: 하단/중간/최상단 3종 상품가 표시 (기존 프리셋 재사용)
- 고객에게 iPad 화면을 그대로 보여주며 상담하는 용도 → 큰 글씨, 표 형태
- 계산 결과는 카드에 스냅샷 저장 (계산 시점, 적용 요율 포함)

## ④ 메모 — 스티커형 손글씨 주석 (포스트잇 패턴)
- 매물카드 화면 어디서든 [+메모] → 화면 중앙에 손글씨 캔버스 확대 오픈
- **펼침 상태 (작성 모드)**
  - Apple Pencil 필기: perfect-freehand 등 필기 전용 렌더링 (signature_pad보다 자연스러운 획)
  - 팜 리젝션: Pencil 사용 감지 시 `pointerType === 'pen'`만 획으로 인식, 터치는 스크롤/제스처로
  - 도구: 펜 색 2~3종, 지우개(획 단위), 전체 지우기, 텍스트 입력 모드 전환(키보드 메모 겸용)
- **접힘 상태 (스티커)**
  - [완료] 시 축소 애니메이션 → 매물카드 위에 스티커로 부착, 필기 내용이 미니어처 썸네일로 표시
  - 드래그로 카드 내 자유 배치 (위치 저장), 카드당 여러 장 가능
  - 탭하면 다시 확대 → 이어서 작성/획 단위 수정 가능
- **저장 구조 (핵심)**: PNG 스냅샷만 저장하지 않고 **획(stroke) 데이터를 JSON으로 저장** → 재편집 가능
  - memos 테이블: `strokes (jsonb)` + `thumbnail_path` (접힘 표시용 PNG) + `pos_x, pos_y` (스티커 위치) + `updated_at`
  - 저장 시점: [완료] 버튼 + 30초 자동 저장 (작성 중 이탈 대비)

## ⑤ 계약 및 결제 → 계약서 자동 주입
- 매물카드에서 [계약 및 결제] 버튼 → 기존 계약서 입력 화면으로 이동
- **자동 주입 필드**: 상호, 업종, 사업자등록번호, 소재지 (매물카드 → 계약서 폼)
- 사용자가 할 일: 광고 상품 버튼 선택(하단/중간/최상단) → 개시일 확인 → 고객에게 전달
- 계약 완료 시 contracts 레코드에 customer_id 연결 (카드에서 계약 이력 조회 가능)

## ⑥ 이후 단계 — 기존 스펙 그대로
- 계약서 설명 → 자필 "들었음" → 성명·서명 → PDF 생성 → Storage 저장 → 공유 시트 → 바로결제 페이지 오픈
- 변경 없음 (CONTRACT-APP-SPEC 섹션 3~6 참조)

## 데이터 모델 추가 (Supabase)
```
customers
- id (uuid), phone (unique, key), name, type(양도자/임차인/기타), created_at

listings (매물카드)
- id, customer_id (FK)
- 상호, 업종, 사업자등록번호, 소재지
- 보증금, 월세, 권리금, 관리비
- updated_at (최근 상담순 정렬용)

calculations (수수료 계산 스냅샷)
- id, listing_id (FK), 권리금수수료율, 계산결과(jsonb), created_at

memos (스티커 메모)
- id, listing_id (FK)
- type (handwriting/text)
- strokes (jsonb — 획 데이터, 재편집용), content (text 메모용)
- thumbnail_path (접힘 스티커 표시용 PNG, Storage)
- pos_x, pos_y (카드 내 스티커 위치)
- created_at, updated_at

contracts (기존 테이블에 컬럼 추가)
- + customer_id (FK, nullable — 단독 계약도 허용)
```
- RLS: 전 테이블 소유자 단독 접근 (개인용 앱)

## 개인정보 유의
- 고객 전화번호·사업자번호를 개인 Supabase에 저장하게 됨 → Storage/DB 비공개 필수, 앱 PIN 잠금 필수
- 회사 고객 데이터의 개인 인프라 보관에 대한 사내 정리(퇴사 시 데이터 이관/파기 포함)는 별도 판단 사항

## 구현 순서 제안 (Claude Code 지시 단위)
1. 탭 내비게이션 + PIN 로그인 골격
2. 인트라넷 iframe 테스트 → 결과에 따라 임베드 or 새 탭 버튼
3. customers/listings 스키마 + 매물카드 CRUD (스키마 변경은 협의 후 적용)
4. 수수료 계산 화면
5. 메모 (텍스트 → 손글씨 순)
6. 매물카드 → 계약서 자동 주입 연결
