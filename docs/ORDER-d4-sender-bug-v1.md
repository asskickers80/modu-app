# ORDER-d4-sender-bug-v1 — D4 발신자 판정 오류 + 가이드 단계 오점등 + 문의 지표

## 증상 (실기기 재현, 맥북=문의자 / 폰=양도인)
1. 메시지 스레드에서 양도인이 보낸 답장이 양쪽 화면 모두 "문의자" 발화로 표시됨.
   (문의자 12:07 발신 → 양도인 12:22 답장 → 둘 다 좌측 '문' 아바타로 렌더)
2. 양도인 홈 가이드가 5단계(문의 수신)→6단계(답장) 점등 없이 바로 "협의 진행 중" 표시.
3. 홈 "가게 지표·문의 알림"의 문의 칸이 실문의 발생에도 "준비중" 고정.

## 조사 항목 (수정 전 원인 보고 먼저)
1. 메시지 발신자 저장·판정 구조: sender를 뭘로 저장하는가(device_id? role? conversation 상의 위치?).
   렌더 시 "내 메시지" 판정 로직이 어디에 있고, 소유권 사건과 같은 device_id 단독 판정인지.
   저장 자체가 잘못됐는지(DB에 sender가 문의자로 기록) vs 렌더 판정만 잘못인지 SQL로 구분해 보고.
2. 가이드 단계 판정: "협의 진행 중"을 점등시킨 이벤트가 무엇인지.
   negotiating status 전환과 문의 수신이 혼동된 것인지, 단계 판정 쿼리가 실데이터를 보는지.
3. 문의 지표: 준비중 하드코딩인지, 카운트 쿼리가 있는데 0을 반환하는지.

## 수정 지시 (원인 보고 후 진행)
1. 발신자 판정을 ownership.js처럼 단일 함수로 추출. 저장은 conversation 참가자 기준(문의자/소유자)으로
   명확히 기록하고, 렌더는 "현재 뷰어가 그 참가자인가"로 판정. 복제 금지.
   기존 오염 데이터(잘못 저장된 sender)가 있으면 SQL 제시 후 멈춤.
2. 가이드: 5단계=첫 문의 수신(DB 관찰 가능), 6단계=소유자 첫 답장(DB 관찰 가능)으로 판정 재정렬.
   "협의 진행 중"은 negotiating status 전환 시에만.
3. 문의 지표: 실 문의 수 카운트로 교체(0이면 0 표시). 조회·관심은 준비중 유지.

## 테스트
Playwright: 2참가자 시나리오(문의자 발신→소유자 답장) 후 양쪽 뷰에서 좌/우 정렬·아바타 판정 검증.
가이드 5→6단계 점등 순서 검증. 실데이터 참·거짓 양쪽 확인 원칙 적용.
