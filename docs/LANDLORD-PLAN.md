# 임대인(Landlord) 적용 계획

> 작성일: 2026-07-03  
> 이 문서는 오늘(07-03) 양도자(seller)에 적용한 작업들을 임대인에 어떻게 적용할지 정리한 계획·질문 문서다.  
> 임대인 코드는 이 문서에 없다. 코딩 전 대표님 결정이 필요한 항목을 먼저 정리한다.

---

## (a) 양도자와 동일하게 복제 가능한 것

아래 패턴은 임대인 흐름에 그대로 이식할 수 있다. 로직 구조가 같고 파일명·변수명만 바꾸면 된다.

| 양도자 작업 | 임대인 복제 대상 | 비고 |
|-------------|-----------------|------|
| `listings.device_id` 컬럼 + 저장 시 `getDeviceId()` 기록 | `landlord_listings` 테이블(또는 동일 `listings`)에 `device_id` 컬럼 추가 + E1pStep5 저장 시 기록 | Supabase SQL 1줄, E1pStep5 코드 1줄 |
| A7SellerDashboard → `supabase.from('listings').eq('device_id', myId)` 패턴 | A7LandlordDashboard → 같은 패턴으로 내 상가 목록 조회 | useEffect 구조 동일 |
| `src/lib/completeness.js` — `calcScore` + `listingToScoreInput` | 임대인용 `calcScoreLandlord` + `landlordListingToScoreInput` 추가 | **점수 항목은 다름** — 아래 (b) 참고 |
| A7 완성도 숫자 실연결 방식 | A7LandlordDashboard 완성도 숫자도 동일 방식으로 실연결 | 하드코딩 값 → 실계산값 교체 |
| E1Step5 중복 제출 방어 (`isSubmitting` ref) | E1pStep5에도 동일 패턴 적용 | 현재 E1pStep5에 방어 로직 있는지 확인 필요 |
| 진입 가드 (`!data.aiDraft` → 가드 화면) | E1pStep3·E1pStep5에도 동일 가드 확인 | 현재 적용 여부 확인 필요 |

## (b) 임대인 특성상 다르게 설계해야 하는 것

### 핵심 전제: 임대인은 고령·디지털 취약 계층 비중이 높다

양도자(가게 파는 자영업자)는 스마트폰에 익숙하고 정보 입력 동기가 강하다.  
임대인(건물주·상가 소유주)은 연령대가 높고 앱 사용 경험이 적을 가능성이 높다.  
**"최소 마찰"이 핵심 설계 원칙이다.**

---

### ① 등록 단계 수 — 5단계 그대로 복사 금지

양도자 E1은 5단계 (기본정보 → AI 초안 → 검수 → 사진 → 최종 공개).  
임대인 E1p도 현재 5단계 구조이나, 이 흐름을 임대인 사용자가 완주할 수 있는지 재검토 필요.

- 양도자 E1의 AI 초안(E1/2) 단계는 Gemini 로딩 대기가 있음 → 고령 사용자가 "멈춘 것"으로 오해할 수 있음
- 검수(E1/3) 단계의 "그대로 / 수정" 칩 UI는 직관적이나, 블록 수가 많으면 피로감 발생
- **제안**: 임대인은 3단계(기본정보 → 사진 → 공개) 또는 2단계로 대폭 단순화 검토

---

### ② AI 초안 — 임대인에게 필요한지 재검토

양도자 AI 초안: 권리금·업종·운영이력을 기반으로 설득력 있는 매물 설명 생성 → 효과 명확.  
임대인 AI 초안: 보증금·월세·위치 정보로 상가 소개 생성 → 효과가 양도자보다 낮을 수 있음.

- 임대인 매물 설명은 "층·면적·보증금·월세"가 핵심 팩트이고 문장형 설명 필요성이 낮음
- AI가 생성한 초안을 임대인이 "검수"하는 단계는 고령 사용자에게 혼란 유발 가능
- **검토 방향**: AI 초안을 없애거나, 있더라도 "자동 완성됐어요, 바로 공개할게요" 방식으로 단축

---

### ③ 완성도 점수 항목 — 양도자와 다름

양도자 `calcScore` 항목: 주소·상호·면적·보증금+월세·권리금·양도방식·검수항목 3개+·사진·매출증빙 (총 100점)

임대인 완성도는 다른 항목으로 설계해야 한다. 예시:

| 항목 | 배점(예시) | 이유 |
|------|----------|------|
| 주소 | 20 | 필수 |
| 면적·층 | 10 | 핵심 스펙 |
| 보증금 + 월세 | 20 | 핵심 거래 조건 |
| 관리비 | 5 | 중요 부대비용 |
| 사진 (외관·내부) | 15 | 공실 상가 사진이 핵심 |
| 입주 가능 시기 | 10 | 창업 준비자 관심 1위 항목 |
| 업종 제한 여부 | 10 | 건물주 허용 업종 |
| 건물 정보 (준공연도·주차) | 10 | 신뢰도 상승 요인 |

→ 항목과 배점은 대표님 결정 후 `calcScoreLandlord()` 함수로 구현

---

### ④ 연락처 교환 패턴 — 동일하나 맥락이 다름

양도자↔양수자: 권리금 협상이 목적 → 연락처 교환 전 진지도 확인 중요  
임대인↔임차인: 임대차 계약이 목적 → 보다 격식 있는 첫 접촉이 기대됨

- DM 우선 + 연락처 교환 요청→수락 원칙은 동일 적용
- 단, D4LandlordChat의 UI 톤을 양도자 D4Chat보다 더 정중하게 조정 검토 (현재 동일 구조)

---

### ⑤ 사진 — 양도자보다 중요도 높음

공실 상가의 경우 사진이 없으면 문의가 거의 없음.  
임대인 매물 등록에서 사진 미첨부에 대한 soft 경고를 양도자보다 더 강하게 표시하는 것이 적절.

---

## (c) 코드 건드리기 전 대표님이 결정해야 할 질문 목록

아래 질문들에 답이 나와야 임대인 흐름 코딩이 시작될 수 있다.

### 등록 흐름

1. **E1p 단계 수를 줄일 것인가?**  
   현재 5단계(양도자 복사 구조) → 임대인에게 맞게 3단계 또는 2단계로 줄일지 결정 필요.  
   줄인다면: 어떤 단계를 없앨 것인가? (AI 초안 제거? 검수 제거?)

2. **AI 초안(Gemini)을 임대인 등록에도 넣을 것인가?**  
   넣는다면 기존 E1pStep2 구조 유지.  
   안 넣는다면 E1pStep2를 단순 미리보기 또는 바로 사진 단계로 교체.

3. **임대인 완성도 점수 항목과 배점을 확정해 주세요.**  
   위 (b)-③ 예시 참고. 대표님이 최종 항목·배점 결정 → 그대로 `calcScoreLandlord()` 구현.

### 데이터 구조

4. **임대인 매물을 어디에 저장할 것인가?**  
   방법 A: 기존 `listings` 테이블에 `listing_type = 'landlord'` 컬럼 추가해서 함께 저장  
   방법 B: 별도 `landlord_listings` 테이블 생성  
   → 방법 A가 단순하지만 필드 구조 차이가 크면 B가 나을 수 있음.

5. **임대인에도 `device_id` 기반 소유권 식별을 쓸 것인가?**  
   로그인 없이 "내 상가" 조회를 하려면 `device_id` 방식 필요.  
   동일 패턴 적용 가능 — 단, 기기 변경 시 소유권 소실 문제는 양도자와 동일하게 존재.

### UX 방향

6. **임대인 대시보드(A7LandlordDashboard)에 "내 공개 상가" 카드가 필요한가?**  
   양도자 대시보드에 방금 추가한 것과 동일한 패턴.  
   임대인이 복수 상가를 등록할 경우 목록 표시 방식도 결정 필요.

7. **고령·디지털 취약 계층을 위한 UI 특별 처리를 넣을 것인가?**  
   예: 글자 크기 키우기, 단계별 진행바 대신 "1/3 단계" 텍스트 강조, 단계마다 "다음" 버튼 외 다른 선택지 없애기 등.  
   현재 A7LandlordDashboard는 양도자 대시보드와 거의 동일한 구조.

대표님 결정이 완료되면 아래 순서로 진행 예정:
