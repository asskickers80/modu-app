## 3. 사용자 카테고리 — ⭐ v16 개정 (7개 → 6개)

### 3-1. ⭐ 핵심 변경: '양수자'+'가맹희망자' → '창업 준비'로 통합

**창업하려는 사람은 결국 한 묶음.** 별도 카테고리로 떼어두니 A2에서 🚀(양수자)·🍔(가맹희망자) 칩이 따로 놀았음. 통합하면 한 카테고리 안의 **2×2 사분면**으로 자연 정리됨.

```
🚀 창업 준비 (A2의 한 카테고리)
│
├─ 개인 직영
│   ├─ 신규 창업 → 빈 점포·상가 찾기
│   └─ 인수 창업 → 양도 매물 인수 (= 기존 '양수자')
│
└─ 프랜차이즈 창업
    ├─ 신규 가맹 → 본사 브랜드 선택 (= 기존 '가맹희망자')
    └─ 인수 창업 → 기존 가맹점 인수 (양도매물 ∩ 프랜차이즈)
```

- **A3에서 방식(직영/프랜차이즈)만 묻는다.** 피드 카드 종류를 가르므로 필수. "둘 다 보고 싶어요" 한 자리.
- **신규/인수는 묻지 않는다.** 본인도 안 정함("좋은 자리 있으면 새로, 괜찮은 양도매물 있으면 인수"). 행동으로 갈리게 둠(큐레이션 철학 13-6절).
- 멀티프로필로 억지로 잇던 '프랜차이즈 인수 ↔ 기존점 인수' 합류점이 한 카테고리 안 사분면으로 자연 통합.

### 3-2. 최종 카테고리 (B2C 5개 + B2B 6분류)

```
B2C (무료, critical mass)
① 양도자
② 창업 준비 ⭐ (양수자+가맹희망자 통합, 2×2: 직영/프랜차이즈 × 신규/인수)
③ 임대인·상가 소유자 (현재 소유주 한정)
④ 운영 중 자영업자 ⭐ (비전 핵심)
⑤ 그냥 구경
⑥ (v2) 구직자

B2B (유료, 기업회원 6분류 — 가입 단위)
1. 중개·컨설팅 / 2. 시설 제공 / 3. 세무·회계·법무 / 4. 금융 / 5. 가맹 본사 / 6. 서비스(캐치올)
```

### 3-3. 비즈니스 모델 본질
```
점포라인: 양도자(약자) → 회사 수익 (호구 구조)
모두:    자영업자(약자) 무료 → 기업 유료 → 회사 수익
1단계 수익: 기업회원 구독 + 매칭 수수료 + B2C 프리미엄
2단계 수익: 데이터 인프라 사업 (Bloomberg 패턴)
```
- **B2C 프리미엄**: AI 처리 속도·처리량 + AI 생성 깊이 + 노출 경쟁 참가권
  - 파는 쪽(양도자·임대인·기업회원) = 노출 경쟁
  - 사는·쓰는 쪽(창업준비·운영중) = 추천의 질·AI 도구·속도 (노출 아님)
- **광고 수익 사용자 정산 없음.** 결제는 오직 사용자→모두 방향만.

### 3-4. 운영 원칙 (Don'ts)
1. No 영업 푸시 / 2. No 책임형 의뢰 / 3. No 광고 수익 사용자 정산
4. **No 노출 순위 직접 판매** (광고는 별도 라벨로만)
5. **No 수요자 폭격** — AI 정제·선별한 수요만. Push도 이중 게이트 필수(9-6절).

```
  👀 그냥 구경 왔어요
  (v2) 💼 일자리를 찾고 있어요
  ※ 기존 🍔 프랜차이즈 칩은 사라짐 → 🚀 창업 준비로 흡수
```

### 5-1. A3 카테고리별 핵심 질문

```
① 양도자
   Q1. 업종?  Q2. 위치?  Q3. 양도방식? [바닥권리/영업양도/고민중]ⓘ

② 창업 준비 ⭐ (통합)
   Q1. 어떤 방식? [내 브랜드로(직영) / 프랜차이즈로 / 둘 다 보고 싶어요]  ← 피드 종류 결정
   Q2. 어느 지역?  Q3. 예산?
   ※ 신규/인수는 안 물음 — 피드 행동으로 갈림
   ※ 프랜차이즈 선택 시에만 본사 리스크가 카드에 따라붙음

③ 임대인
   Q1. 상가 위치?  Q2. 상태? [공실/매각/둘다]  Q3. 몇 개? [1/2~3/4+]

④ 운영 중
   Q1. 업종?  Q2. 위치? ('온라인·무점포' 포함)  Q3. 매출 관리? [연동/수동/아직]

⑤ 기업회원 (6분류 공통)
   Q1. 어떤 기업회원?  Q2. 활동 지역?  Q3. 사업자등록증 인증 → 보라색 모드

⑥ 그냥 구경 — 질문 없음 → 카드 피드 직행
```

**공통 구조**: Q1·Q2는 공용 부품(업종·지역), Q3만 카테고리색.

### 5-2. A4 가입 방식
계정은 하나, 입구만 여럿. 카카오(최상단)/네이버/Apple(iOS 필수)/휴대폰. 이메일은 PC 웹·기업회원용으로 한 단계 접음.

### 5-3. [F] 공통 인증 게이트 — 토스 패턴
민감한 것이 연결·노출되는 순간 튀어나오는 트리거 기반 게이트.
```
트리거: 매출·POS·세무 연동 / 매물 공개 / 돈이 오갈 때 / 사업자 정보 연결
그 전까지: 둘러보기·콘텐츠·질문·관심표시는 인증 없이 자유
[개인 B2C] 본인인증 1회
[기업 B2B] 본인인증 + 사업자등록증 인증 2단계 → 보라색 모드
※ 연락처 교환도 일종의 트리거 게이트
```

### 5-4. 양도자 양도방식 분기
```
가. 바닥권리(자리 양도형): 영업 안 넘김. 자리값+시설 잔존가. → 입지·시설·빠른 회전 강조
나. 영업 양도(권리금형): 영업 통째로. 매출·단골 포함. → 매출증빙·수익성·진지한 양수자 강조
```

- 창업준비의 인수 선호와 자물쇠-열쇠로 맞물림.

## 9. ⭐ 차별화 포지셔닝 + 매칭 엔진

### 9-1. 핵심 한 줄
> "모두는 자영업자의 생애를 함께하는 OS이며, 가장 강력한 무기는 지원 업체 인프라다."

### 9-2~3. 캐시노트 대비 (⭐ v16 경쟁 분석 갱신)
**캐시노트 업체찾기 실측 결과**: 자영업자가 검색→견적문의→카톡 알림 받는 **수요자용 디렉토리**. 업체 추천은 사람이 도와줌. 약관에 업체 정보 등록·제휴 조항은 있으나, **업체에서 본격 수익을 내는 구조가 아님**(수익은 자영업자 구독·마켓·금융). 업체 '영업 상황판'·능동 영업(Push) 없음.
```
결론: 무게중심이 반대.
 · 캐시노트 = 자영업자용 종합 포털, 업체찾기는 곁가지
 · 모두 = 업체↔자영업자 매칭 '자체가 본체' + 기업회원 주력 수익
 · 모두의 양방향 AI 정제 매칭 + 영업 상황판은 시장에 빔
 · 단 캐시노트는 190만 데이터로 마음먹으면 빨리 옴 → 해자는 '속도 + 업계 관계망 + 업체를 주인공으로 두는 설계 깊이'
```

### 9-4. ⭐ 양면 매칭 통합 원칙
모두는 하나의 양면 매칭 생태계. 파는 쪽(양도자·임대인·기업회원 6분류)과 사는·쓰는 쪽(창업준비·운영중)이 전부 같은 메커니즘(추천 피드 카드 + 노출 3층 + DM)으로 엮임.

### 9-5. ⭐ AI 정제 수요 매칭 엔진 — PULL
```
[수요 발생] 행동·검색·직접 요청
   ↓
[모두 AI 정제] ★우리 힘★ 진지도·적합도 거름
   ↓
[업체 알림] 적합 업체 ①알림함 "이런 요청, 가능?" (마구잡이 ❌)
   ↓
[관심 업체 DM] 여러 업체가 각자 수요자와 1:1 개시
   ↓
[병렬 1:1] 수요자 ↔ 업체A/B/C (서로 못 봄 = 비공개는 DM 구조상 저절로)
   ↓
[매칭=연락처 교환] 수요자 "이 업체로" → 업체 "수락" → 동시 공개 + 성사
```
- 헤이딜러: 방향만 차용("수요자 가만히, 공급자 움직임"). 입찰·정원제·줄세우기는 버림.
- **비공개는 설계 변수 아니라 DM 구조의 부산물.** 서비스는 가격 단일화 불가 → 1:1 DM이 본체.

### 9-6. ⭐⭐ PUSH 루트 (v16 신규) — 업체가 먼저 거는 영업
> 기존 Pull(수요→업체)에 더해, 업체가 먼저 수요자에게 거는 길. **수요자 메시지 폭탄(Don't 5) 방지가 핵심.**
```
업체 "영업하고 싶다" (분류·지역 선택, 발신 쿼터)
   ↓
[이중 게이트 — 둘 다 통과해야 발신]
   게이트 1 · 수요자 설정: "이런 제안 받을게요" 분류별 ON 한 것만  ★시작 중심(명시적 동의)★
   AND
   게이트 2 · AI 적합도: 행동·상황으로 '받을 만한지' 추정 → 발신 창 개방 (보조)
   ↓
[수요자] 제한된 first DM 수신 → 무시하면 끝(읽씹=거절) → 관심 시 1:1
   ↓
[이후 Pull과 동일] 1:1 → 연락처 교환
```
- **게이트는 못 판다**: 발신 양·속도만 유료, 폭탄 방지선은 비매품. 발신 총량 상한(프리미엄도 적용).
- 게이트 느슨해지는 순간 '수요자 보호' 정체성 무너짐 → 보수적 유지.
- 구현 이월: 게이트2 적합도 기준·발신 쿼터 수치.

### 9-7. ⭐⭐ 매칭 성사 vs 연락처 교환 — B2C/업체매칭 분리 (v16 핵심 정정)
> **대표님 정정**: 매물 거래(B2C끼리)엔 '매칭 성사'·'이 사람으로 정하기'가 구조적으로 존재할 수 없음. 동시에 여러 곳 연락·현장방문 가능, 계약 전엔 아무것도 확정 안 됨. 앱은 '성사'를 선언할 자리가 아님(계약은 앱 밖 현실).

```
                  연락처 교환(요청→수락)   '매칭 성사' 선언
B2C 거래                                    
(양도·임대·창업준비)    ✅ 있음              ❌ 없음 (계약 전 미확정)
업체 매칭                                    
(수요자↔기업회원)      ✅ 있음              ✅ 있음 (수수료·KPI 기준점)
```
- **차이 이유**: 업체 매칭은 모두가 정제한 수요를 연결한 것이라 '성사'가 수익·KPI 기준점. B2C 매물은 사람끼리 만나는 일이라 잴 수도 이유도 없음.
- **연락처 교환 = 요청→수락 (한 동작)**: 양쪽 동시 공개, 일방 캐가기 불가, 거절=방 나가면 끝. 누르기 전 모달로 "연락처 공개" 미리 고지(사고 방지 급소).
- ⭐ 양도자·임대인은 '받는 쪽' — 모든 문의 성실 응대, '정하기' 버튼 없음.
- ⭐ 창업준비 사분면별 거래 상대 다름:
  - 직영·신규 → 임대인·중개 (B2C, 성사 없음)
  - 직영·인수 → 양도자 (B2C, 성사 없음)
  - 프랜차이즈·신규 → 가맹 본사(기업회원) (성사 있음)
  - 프랜차이즈·인수 → 양도자(가맹점주) (B2C, 성사 없음)

## 13. ⭐ 매물·프로필 등록 흐름 + 공공데이터 + 기술 스택

### 13-1. E1 양도자 매물 등록 — 5단계 (MVP 핵심①)
```
[1단계] 기본 팩트 — 주소(→건축물대장 자동)·상호·층·면적·임대조건·희망권리금·양도방식ⓘ·(영업양도시)월매출
[2단계] AI 초안 — 지오코딩→상권API·웹검색·매출비교→LLM 설명 초안 (사실=단정/추정=참고 톤 분리)
        ※ 무료=최소 정보 / 프리미엄=차별화 생성
[3단계] 검수·수정·공개 선택 ★차별점★ — 항목별 [그대로][수정][공개안함], 검수 로그 적재(학습 연료)
[4단계] 사진·증빙 보완
[5단계] 완성도 게이지 → [매물 공개] → [F] 인증 게이트 → 대시보드 ④ 연동
```

### 13-2. 노출 3층 구조 (파는 쪽 공통)
```
1층 무료: AI 최소 정보, 기본 노출
2층 프리미엄(월·저렴): AI 차별화 생성 + 완성도 기반 노출 경쟁 + 매칭 확률↑
3층 광고(별도, 차후): 2층 경쟁 초월, 광고 라벨 명시, 줄세우기와 분리
적용: 양도자·임대인·기업회원(6분류) ✅ / 창업준비·운영중 ❌(추천질·AI도구·속도가 프리미엄)
순위 원칙: 완성도·정확도·최신성 + 적합도. 프리미엄=AI 생성 깊이·매칭 양/속도. 순위 직매 ❌.
```

### 13-3. 공공 데이터 — "발급 아닌 열람·추출"
> 공공 데이터는 "발급"(승인·과금·본인인증) 아니라 "열람·추출". 주소 한 줄로 면적·층·시세·등기 자동. 사실은 무료, AI 해석은 프리미엄.
```
건축물대장: 국토부 건축HUB API, 주소만 (무료·인증 불필요) → 면적·층·구조 자동
실거래가: 공공데이터포털 API (무료·키 발급) → 시세 비교
등기 핵심: 등기정보광장/CODEF '열람' (발급❌) → 근저당·선순위·압류
무료=사실 표시 / 프리미엄=AI 위험해석 ("보증금 안전 구간")
```
- 구현 검증: 등기 '열람' 소스(등기정보광장 vs CODEF)·호출 절감("공개 누르는 시점 1회 조회").

### 13-4. AI 정확도 향상
가. 표본 증가(자동·확실) / 나. 검수 로그→교정(진짜 무기, 적재 설계 필수) / 다. 파인튜닝(1만+ 후)

### 13-5. E1' 임대인 등록 — 양도자 5단계 뼈대 100%, 4곳+등기 변형
```
1. 1단계 분기: 양도방식 → 임대/매각/둘다
2. 2단계 ③: 매출위치 → 임대시세/수익률(캡레이트) + 등기 자동열람
3. 시세 공개: 방어적 → 전략적 (AI "공개시 문의↑" 코칭)
4. 4단계 증빙: 매출증빙 → 도면 (등기는 2단계 자동, 뗄 필요X)
5. 5단계 귀결: 매물 완성도 → 자산별 카드 (여러 개면 등록 반복)
+ 고령 임대인 UX: 주소만으로 자동, 타이핑 최소, (v1.5)대리등록·전화상담
```

### 13-6. 창업 준비 프로필 — ⭐ v16 통합 (큐레이션형, 2×2)
> **등록하지 않고 본다.** 추천 피드가 전경, 프로필 카드는 행동으로 만들어지는 배경.
```
1단계: A3에서 최소 프로필 (방식·지역·예산). 심화: 어떤 인수?·단계·예산
2단계: 추천 피드 생성 ★첫 화면 주인공★
       = 2×2 사분면 (직영/프랜차이즈 × 신규/인수)
         직영·신규 → 빈 점포 / 직영·인수 → 양도매물
         프랜차이즈·신규 → 본사 브랜드(⚠리스크) / 프랜차이즈·인수 → 가맹점 양도(⚠리스크)
       행동(본·찜·체류·문의)이 신호 → 엔진이 추천 정밀화 + 신규/인수 갈림
3·4·5단계 증발 (검수·사진·완성도 없음)
노출 폐기: 사는 쪽은 노출 안 함. 프리미엄=추천질·신규알림·AI분석 깊이.
문의는 DM (사분면별 상대 다름, 9-7절). 프랜차이즈에만 본사 리스크 동반.
```

### 13-7. ⭐ 기업회원 노출 페이지 (E1'') — A장·B장 완료

**관점: 첫 페이지 = "회사 소개"가 아니라 "영업 상황판".**

**진입 = 사업자등록증 1장** (번호만으론 부족):
```
국세청 API: 상태조회(번호만)=휴폐업·과세유형뿐 / 진위확인(번호+α)=대조만
→ 흐름: 등록증 촬영 → AI OCR(상호·업태·종목·소재지·개업일=업력)
        → 국세청 진위확인+휴폐업 확인 → 검증 배지
        → 업종코드 → 12분류 자동 귀속 → 보라색 모드
```

**⭐ 노출 분류 12개 (v16 확정, 9개→12개):**
```
🔨 인테리어·간판        → 시설
🧰 설비·시공            → 시설
💳 결제·IT·보안 ⭐신규   → 서비스  (POS·키오스크·CCTV·통신, 배리어프리 의무화 수요↑)
📦 식자재·물품          → 서비스
🧹 청소·위생            → 서비스
📣 마케팅·홍보          → 서비스
🚚 배달·물류 ⭐신규      → 서비스
🧑‍🍳 구인·인력            → 서비스
🧾 세무·회계·법무·노무   → 세무·회계·법무
💰 금융·보험 ⭐+보험     → 금융
🤝 중개·컨설팅          → 중개·컨설팅

```
- 가입은 6분류로 굵게, 노출·매칭은 12개로 잘게 (층위 분리). 캐시노트 등 현실 데이터로 검증.

**노출 페이지 5블록 (업종 무관 공통):**
```
① 한 줄 정체성 [검증·자동] — "지역+업력+분류". 수정불가. 매칭 1차 필터.
② 이럴 때 부릅니다 [매칭 트리거] ★심장★ — 호출상황 3~5개. = 추천카드 카피+수요 매칭 키.
③ 무엇을 해결합니다 [해결 3쌍] — 문제→해결. 공통축=돈/신뢰/시간.
④ 믿을 근거 [신뢰 증거] — 검증배지·업력·실적·후기·포트폴리오.
⑤ 견적·문의 [DM] — 번호 비공개. 매칭 성사 시 연락처 교환.
※ "이 물건이 좋다"(양도자)가 아니라 "당신 상황에 내가 맞다" → 자랑(③④)보다 상황매칭(②)이 위로.
```

**AI 생성 강도 = '강하게' + 안전장치:**
```
강하게: 등록증+API+소스로 거의 완성, 사용자는 확인·수정만
안전장치: 사실/생성 톤 분리 / 블록 단위 인라인 수정·"다시 써줘" / 거짓·과장 차단(소스 없이 강점 지어내기❌)
자동완성: 소스 0개여도 ①②③⑤+④사실 완성. 소스(회사소개서·홈페이지·사진·음성) 줄수록 ③④ 풍부.
```

**무료 vs 프리미엄 매칭 차등:**
```
유료가 사는 것 = 기회의 양·속도·범위 (더 많이·먼저·넓게 닿음)
유료가 사면 안 되는 것 = 같은 수요 안 '줄세우기 1등' 직매
무료: 내 12분류·지역·표준속도 알림
프리미엄: 가.양(인접분류+넓은지역) 나.속도(먼저) 다.노출(피드 자주·위, 광고라벨) 라.정보(놓친수요 선제)
         + 페이지 살(③④·포트폴리오 = 노출3층 2층)
```

**구현 이월**: ③살붙이기 프롬프트 / 업종코드→12분류 룩업(다중업종 우선순위) / ②↔매칭키 연결 / 신규3분류 ②③ 표준세트 / 매칭 차등 가중치.

### ⭐ 통찰 90-95 (v16 신규)

90. **B2C 거래엔 '매칭 성사'가 없다** — 매물 거래(양도·임대·창업)는 동시에 여러 곳 연락·현장방문 가능하고 계약 전엔 아무것도 확정 안 됨. 앱은 '성사'를 선언할 자리가 아니다(계약은 앱 밖). 필요한 건 연락처 교환(요청→수락)뿐. '성사 선언'은 업체 매칭(수요자↔기업회원)에만 — 그게 수수료·KPI 기준점이라서.

91. **양도자·임대인은 '받는 쪽'** — 여러 문의에 전부 성실 응대. '이 사람으로 정하기' 같은 선택 장치 없음(누가 될지 계약 전 모름). 고르는 쪽(수요자)이 연락처를 '요청', 받는 쪽이 '수락'. 모달 양면은 그대로 살되 B2C는 '연락처 교환', 업체는 '연락처 교환=성사'.

92. **Push 루트 — 업체가 먼저 거는 영업, 단 이중 게이트로 폭탄 방지** — 영업의 본질은 업체가 능동적으로 움직이는 것. 그래서 Pull만으론 부족. 단 무제한 콜드 메시지는 Don't 5 정면 위반 → 모두가 문지기. 게이트1(수요자 명시 동의) AND 게이트2(AI 적합도). 게이트는 못 판다(발신 양·속도만 유료, 방지선은 비매품). 시작은 게이트1 중심으로 보수적.

93. **'양수자'+'가맹희망자' → '창업 준비' 통합** — 창업하려는 사람은 한 묶음. 2×2(직영/프랜차이즈 × 신규/인수)로 정리. A3에서 방식만 묻고(피드 종류 가름), 신규/인수는 행동으로 갈리게 둠(큐레이션 철학). 멀티프로필로 억지로 잇던 합류점이 사분면 내부로 자연 통합. 카테고리 7→6개.

94. **노출 분류 9→12개** — 현실 데이터(캐시노트 13카테고리·창업비용·점포 생애주기)로 검증. 결제·IT·보안 / 배달·물류 독립, 금융에 보험, 프랜차이즈 본사 노출 격상. 모든 점포가 반드시 쓰는 걸 뭉개지 않음. 가입 6분류/노출 12개 층위 분리.

95. **캐시노트는 무게중심이 반대** — 업체찾기는 수요자용 디렉토리(검색·문의·안내)에 머물고, 업체에서 본격 수익을 내거나 업체 영업 상황판·능동 영업(Push)을 갖추지 않았다. 모두는 '업체↔자영업자 매칭 자체가 본체' + 기업회원 주력 수익. 해자는 "아무도 안 함"이 아니라 "본체로 파는 사람이 곁가지로 하는 사람을 이긴다" + 속도·업계 관계망.

96. **방문자 열람 자유 원칙** — 탐색·검색·매물 상세(E2)·기업회원·커뮤니티/이야기 등 모든 공개 콘텐츠는 비로그인 개방. [F] 인증 게이트(5-3절)는 오직 **행동 시점**에만 건다: 문의(DM) 발신 / 찜·저장 / 매물·글 등록 / 마이 등 프로필 필요 기능. 게이트는 차단 페이지로 튕기지 않고 현재 화면 유지 + 가입 유도 시트, 가입 완료 후 원래 지점으로 복귀(returnTo). (원칙은 5-3에 이미 명시 — 코드가 방문자 홈에서 열람까지 막고 있던 것을 ORDER-guest-access-v1에서 교정.)

97. **가이드 문의 단계 = 문의받기·협의시작 2칸 구조** — 진행 가이드의 문의 국면은 두 칸으로 구성한다: (5) 문의받기 = 첫 문의 수신(DB 관찰), 완료 시 **첫 문의 일시 노출**("N월 N일 첫 문의 도착")·탭하면 그 스레드로 딥링크. (6) 협의시작 = 소유자 첫 답장(양방향 대화 성립, ownerReplied)·미답장 문의가 있으면 "답장을 기다리는 문의 N건" 유도(모두 화법)·탭하면 인박스. status='negotiating' 전환은 별도 개념(탐색 뱃지·접힘 카드 전용). 판정은 전부 DB 관찰 가능해야 하고, 발신자 판정은 '문의자(conversation.sender_id) 아닌 발신'으로 device_id desync에 견고하게.

98. **문의 지표 = 새 문의(미확인) 주 표시 + 누적 서브, 읽음 판정 단일 소스** — 홈 '가게 지표'의 문의 칸은 **새 문의(미확인) 수를 최우선**으로 강조(색·크기, 0이면 강조 해제)하고, 누적 전체 수는 비활성 톤 서브("전체 N")로 뒤에 둔다. 탭: 미확인 1건이면 그 스레드 딥링크, 2건 이상(또는 0)이면 인박스. **미확인 판정은 `lib/unread.isUnread`(conversations.receiver_last_read_at 기준) 단일 소스** — 메시지 탭 배지(useHasUnread)·D4 진입 읽음 갱신(markConversationSeen)과 동일 소스라 복제·불일치 없음. 별도 owner_read_at 컬럼 불필요(receiver_last_read_at가 이미 그 역할). "진척도"(구 진지도)는 로직 미구현이라 준비중 유지 — 명칭만 교체(가짜 수치 금지 원칙).
